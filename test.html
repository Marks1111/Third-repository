This will be a test for structure of a git commit

Here is the structure of a proper git commit

type [optional scope]: description

[optional body]

[optional footer(s)]

There are different types that we can use to describe the beginning of the commit

fix/feat/build/chore/ci/docs/style/refactor/test/perf - There are the most common ones

Here is an example of how the type would be committed

fix/feat/build [optional scope]: description

[optional body]

[optional footer(s)]

Here is a how the [optional scope] would be committed

fix/feat/build (Explains the scope of the "change" and it has to be surrounded by parentheses)!: description

[optional body]

[optional footer(s)]

Here is how a BREAKING CHANGE would be commited

fix/feat/build [optional scope]If this commit has any BREAKING CHANGES add a BREAKING CHANGE: or a ! to it: description

[optional body]

[optional footer(s)]

A BREAKING change is a commit that has a footer BREAKING CHANGE: or a ! after the type/scope which introduces a breaking API change to the program, commit, or project

Here is how a description would be committed

fix/feat/build [optional scope]: Usually short primitive message explaining what has been done like add

[optional body]

[optional footer(s)]

Here is the body of a proper commit

fix/feat/build [optional scope]: description

Introduce a request id and a reference to latest request. Dismiss

incoming responses other than from latest request.

[optional footer(s)]
